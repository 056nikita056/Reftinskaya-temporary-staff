# Что сделано

## Авторизация

- Экран входа упрощен: остались только логин и пароль.
- Убран выбор роли при входе.
- Убран выбор фабрики при входе.
- Логотип с экрана входа убран.
- После входа сервер сам определяет фабрику и права пользователя.
- Выход из аккаунта работает через кнопку в профиле и в мобильной шапке.

## Таблицы доступа

Схема доступа приведена к нужной структуре:

```text
users
roles
factories
users_roles
users_factories
```

Таблица `users_roles` связывает пользователей с ролями.

Таблица `users_factories` связывает пользователей с фабриками.

В `users_factories` добавлено поле:

```text
is_primary
```

Оно показывает основную фабрику пользователя. Если у пользователя несколько фабрик, после входа выбирается основная.

## Роли и права

Пользователь может иметь несколько ролей одновременно.

Сервер собирает общий набор прав из всех ролей и возвращает:

```text
access.modules
access.actions
```

Меню строится по `access.modules`.

Действия внутри экранов проверяются по `access.actions`.

Добавлены более точные права:

```text
plans.factory.edit
plans.hr.edit
plans.out.edit
facts.factory.edit
facts.out.edit
```

## Планы

Планы теперь учитывают не одну выбранную роль, а набор прав пользователя.

Текущий этап плана определяется по статусу:

```text
В доработке             -> фабрика
Отправлено              -> HR
Получено / Не утверждено -> аутсорсинг
```

Это исправляет сценарий, когда у пользователя есть сразу `factoryPlanner` и `hr`: на статусе `Отправлено` он работает как HR, а не откатывает план обратно в фабричный этап.

Редактирование полей разделено:

```text
plans.factory.edit -> фабричные поля плана
plans.hr.edit      -> штатные поля
plans.out.edit     -> поля аутсорсинга и распределение людей
```

## Факты

Фиксация фактов тоже работает по правам, а не по одной роли.

Если у пользователя есть оба права, появляется переключатель:

```text
Фабрика / Аутсорсинг
```

Права:

```text
facts.factory.edit -> факт и пояснение от фабрики
facts.out.edit     -> факт и пояснение от аутсорсинга
```

Пояснения нельзя создать без права на выбранную сторону.

## Backend-проверки

Добавлены серверные проверки для `compat`-мутаций.

Теперь backend проверяет права для:

```text
plans
operations
sections
assignments
employees
reservations
housing-dorms
settings
facts
explanations
```

Если права нет, сервер возвращает `403` и список недостающих прав:

```json
{
  "code": "FORBIDDEN",
  "missingActions": ["plans.out.edit"]
}
```

Также исправлены обходы:

- неизвестный статус плана больше не проходит через общий `plans.edit`;
- изменение несуществующей записи возвращает `404`;
- сторона пояснения берется из сохраненного факта, а не из тела запроса.

## Пользователи

Добавлен скрипт создания пользователей:

```bash
npm run create:user -- --login ivan --password pass12345 --name "Иван Петров" --roles hr,factoryMaster
```

Пример с несколькими ролями:

```bash
npm run create:user -- --login multi --password multi12345 --name "Мульти Пользователь" --roles factoryPlanner,hr,factoryMaster
```

Пример с фабрикой:

```bash
npm run create:user -- --login ivan --password pass12345 --name "Иван Петров" --roles hr --factory "Рефтинская"
```

Если у пользователя еще нет основной фабрики, первая добавленная фабрика становится основной автоматически.

## API

Добавлен endpoint:

```text
GET /api/v1/users/me
```

Он возвращает профиль текущего пользователя, роли, фабрики, модули и действия.
